# 느낀점
RAG와 LLM 관련 프로젝트를 첫진행하다.
바이브 코딩으로 진행할 예정이므로, llm, RAG 기술전반에 관련된 기본 개념 습득이 매우 중요하다.
사전에 몇번이나 RAG관련 책을 보았고, 정리하였다.
개념적인 이해를 바탕으로 GEMINI와 기획을 같이 진행하고,
아키텍처를 설계하였다.

복병은 따로 있었다. 주요 이슈는 아래와 같다.
1. 모델을 학습할수 없으므로(환경상) 프롬프트 엔지니어링으로 최대한 커버해야 하는데, 모델의 성능이 기대에 못미치는 것이다.
예를 들어 문제는 "현재진행형" 인데 내가 작성한 영작문을 보고 모델이 "과거형" 이라고 판단하는 경우가 있다던지.
*이 문제는 적절한 프롬프팅으로 해결하였다. 프롬프팅에 기존에 문제와, 유저답변을 같이 넣어서 이걸 기준으로 판단을 내리라고 정의 하였다.


2. 계속 비슷한 유형의 문제를 제공하는 경우이다. 같은 문제를 계속 반복하는 경우도 있었다. 살펴보니 이것은 temperature가 너무 낮게 설정되어 있어서 다양성이 부족했던 것이 원인이었다. 
   
   *0.4 -> 0.8로 상향 조정하여 해결하였다. 또한 프롬프팅을 강화해서 댜앙한 문제를 출제 해달라고 요청했다. "{current_question}" 이라는 파라미터로 직전 문제를 프롬프팅에 넣어서, 이 문제와 다른 문제를 출제 해달라고 요청하였다.
   
3. 문제자체가 이상한 경우이다. 한글로 문제를 제공하는 과정에서 문맥이 말이 안되는 경우가 있었다. 이건 모델 자체의 한계로 판단하고, 모델을 gpt-4o-mini에서 gpt-4o로 상향 조정하여 해결하였다. 비용이 좀 더 들지만, 문제 품질이 확실히 올라갔다.
